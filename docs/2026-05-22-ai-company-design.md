# AI 公司框架 — 设计文档

## 概述

AI 公司是一个将 AI 模型当作"员工"来管理的框架。用户可以配置模型 Key 和 AI Agent，通过 HR Agent 根据项目需求自动组建项目团队，实现"像管理一家公司一样管理 AI 资源"。

## 核心架构

### 四层结构

```
用户接入层
├── CLI 终端（完整命令行接口）
└── 移动聊天 Agent（手机端对话式控制）

控制层
└── 中央调度器（纯程序，不依赖 LLM）
    ├── 命令路由
    ├── 项目生命周期管理
    ├── 调用 HR Agent
    ├── 启动项目协作空间
    └── 进度追踪与控制

资源层
├── Key 池（OpenAI / Claude / Gemini 等模型 Key）
└── Agent 池（AI 员工定义）

项目执行层
├── 项目工作空间（平等协作 · 互相 Review）
└── 监控 Dashboard（实时可视化面板）
```

### 🔒 硬性规定：中央调度器必须纯程序驱动

**这是整个框架最核心的设计约束，不可违反：**

中央调度器的所有决策（命令路由、项目生命周期管理、进度追踪、HR 调用触发、协作空间创建、超时控制、资源调度）**必须由纯代码实现**，不允许任何 LLM 调用参与调度决策。

LLM 只允许在以下两个位置使用：
1. **HR Agent** — 分析需求、调研模型、推荐团队（选人决策本身是分析类任务，允许 LLM）
2. **项目工作空间** — 被选中的 AI 员工生成内容、讨论、Review

**调度器和执行器严格分离**：调度器（程序）决定"谁什么时候做什么"，执行器（LLM/agent）负责"怎么做"。

### 关键设计决策

| 决策 | 选择 |
|------|------|
| 调度方式 | 纯程序调度，LLM 仅用于 HR 分析和内容生成 |
| 协作模式 | 平等协作，选中的 agent 互相讨论和 review |
| HR 确认模式 | 按成本阈值区分，超过阈值需人工确认 |
| 需求输入 | 自然语言描述，HR 自动提取结构化信息 |
| 上下文管理 | **写文档 → 自杀 → 读文档恢复** 干净上下文模式 |

## 上下文管理：写文档 → 自杀 → 恢复

Agent 的每次工作都必须以干净的上下文开始和结束，不允许上下文不断累积膨胀。

### 生命周期

```
Agent 被调度器唤醒
    │
    ▼
读取项目文档（了解当前状态和上下文）
    │
    ▼
执行工作任务（LLM 在干净上下文中工作）
    │
    ▼
任务完成 → 写入项目文档
    │  · 记录做了什么
    │  · 记录当前状态和结论
    │  · 记录未完成事项 / 后续步骤
    ▼
Agent 自杀（结束会话，释放上下文）
    │
    ▼
等待调度器下次唤醒 → 重新读文档 → 继续
```

### 详细规则

**写文档（任务完成时）：**
- 必须更新项目文档，包含：完成内容、当前状态、决策记录、未完成事项
- 文档是项目工作空间的一部分，所有人可读
- 文档格式统一，方便下次读取时快速理解

**自杀（任务结束时）：**
- Agent 主动终止自己的运行会话
- 释放所有上下文窗口资源
- 在 Dashboard 上标记为"空闲"状态

**恢复（再次被调用时）：**
- 调度器创建全新的上下文窗口给 Agent
- Agent 启动时先读取项目文档，而不是继承历史对话
- 相当于"上班先看日报"，而不是"带着上周的脑子继续干"

**好处：**
- 上下文窗口永远不会膨胀 → 降低 Token 消耗
- 避免 LLM 被陈旧历史带偏
- 每次决策都基于最新文档而非累积对话
- 每个会话可独立追踪和审计

## HR Agent 选人机制

```
项目需求（自然语言）
    │
    ▼
步骤 1：需求分析
    │  HR 读取需求 → 提取项目类型、技术栈、复杂度、交付形式
    ▼
步骤 2：模型调研
    │  HR 联网查询各模型官网 → 了解各模型最新能力
    ▼
步骤 3：匹配推荐
    │  HR 对比 Agent 池中能力标签 → 推荐最佳组合 + 理由
    ▼
步骤 4：成本判断
    │  预估 API 消耗 → 低于阈值：自动组建 | 高于阈值：等用户确认
    ▼
中央调度器创建项目空间
```

## AI 员工配置

每个 AI 员工包含以下配置项：

| 配置项 | 说明 |
|--------|------|
| 名称 | 员工名字 |
| 角色描述 | 系统提示词，定义其职责和行为 |
| 绑定模型 | 关联到 Key 池中的具体模型 |
| 能力标签 | 用于 HR 匹配，如 `["React", "Python", "UI"]` |
| 模型参数 | temperature、max_tokens 等 |
| 速率限制 | 每分钟/小时最大调用次数 |
| 最大并发任务数 | 同时可参与的项目数 |

## CLI 命令体系

```
ai-company
├── config                    # 全局配置
│   ├── init                  # 初始化框架
│   ├── show                  # 查看当前配置
│   └── set cost-threshold    # 设置成本阈值
│
├── key                       # Key 池管理
│   ├── add <provider> --key  # 添加模型 Key
│   ├── list                  # 列出所有 Key
│   ├── delete <id>           # 删除 Key
│   └── test <id>             # 测试 Key 可用性
│
├── agent                     # AI 员工管理
│   ├── create                # 创建新员工
│   ├── list                  # 列出所有员工
│   ├── show <name>           # 查看员工详情
│   ├── edit <name>           # 修改员工配置
│   ├── delete <name>         # 解雇员工
│   └── test <name>           # 测试员工响应
│
├── project                   # 项目管理
│   ├── new <description>     # 提交新项目
│   │   └── --mode auto/confirm
│   ├── list                  # 列出所有项目
│   ├── status <id>           # 查看项目详情
│   ├── pause <id>            # 暂停项目

│   └── cancel <id>           # 取消项目
│
└── dashboard                 # 面板
    └── open                  # 打开 Web 监控面板
```

移动端通过聊天 Agent 控制，自动将自然语言映射为 CLI 命令。

## 监控面板（Dashboard）

交互设计原则：**每项数据都可点击下钻**，从宏观概览逐层深入到细节。

### 导航结构

```
Dashboard 首页（总览）
│
├── 项目卡片（进行中: N）
│   └── 点击 → 项目列表页
│       └── 点击某个项目 → 项目详情页
│           ├── 项目信息：目标、进度、创建时间
│           ├── 团队成员列表：各 AI 员工及其状态
│           │   └── 点击某个员工 → 员工详情面板
│           │       ├── 基本信息：角色、绑定模型、能力标签
│           │       ├── 当前任务 & 任务进度
│           │       ├── 对话历史摘要
│           │       ├── 产出列表
│           │       └── Token 消耗统计
│           └── 项目对话时间线
│
├── AI 员工卡片（在职: N）
│   └── 点击 → 全部员工列表
│       └── 点击某个员工 → 员工详情面板（同上）
│
├── API 消耗卡片（今日: $X.XX）
│   └── 点击 → API 消耗详情页
│       ├── 按 Key 汇总：每个 Key 的请求次数 / Token 数 / 费用
│       ├── 按模型汇总：GPT-4、Claude、Gemini 分别消耗
│       ├── 按项目汇总：每个项目的 API 消耗
│       └── 时间维度：今日 / 本周 / 本月 / 自定义
│           └── 点击任意条目 → 查看该 Key/模型/项目的详细调用记录
│
└── 已完成项目卡片（N）
    └── 点击 → 已完成项目列表
        └── 点击某个项目 → 归档项目详情（只读查看）
```

### 交互行为

| 操作 | 行为 |
|------|------|
| 点击概览卡片 | 进入对应的列表/详情页面 |
| 点击项目 | 进入项目详情，查看团队、进度、产出 |
| 点击员工 | 弹出员工详情面板（非跳转），可查看任务和消耗 |
| 点击 API 消耗 | 进入消耗分析页面，多维度下钻 |
| 返回按钮 | 逐级返回上一级 |
| 面包屑导航 | 显示当前位置，支持直接跳回任意层级 |

## 技术栈建议

基于 fork Paperclip + 借鉴 Mesa 的策略：

| 层 | 技术 | 来源 |
|----|------|------|
| 后端 | Node.js / TypeScript | Paperclip 现有 |
| 前端 UI | React + Vite | Paperclip 现有 |
| 数据库 | PostgreSQL（本地用 PGlite 嵌入式） | Paperclip 现有 |
| CLI | TypeScript CLI | Paperclip 现有 + 改造为你的命令设计 |
| 移动入口 | Telegram Bot（参考 Mesa） + 移动聊天 Agent | 新建 |
| 调度引擎 | Paperclip Heartbeat + DB 级原子锁（纯程序） | Paperclip 现有，强化纯程序保证 |
| 适配器系统 | Paperclip 现有 8+ 适配器 | Paperclip 现有 + 按需扩展 |

## Paperclip Fork 开发策略

### 保留的 Paperclip 子系统
- Identity & Access（身份与访问）
- Org Chart & Agents（组织架构 — 需改造为池分离管理）
- Work & Tasks（任务管理）
- Heartbeat Execution（心跳调度 — 确认纯程序）
- Governance & Approvals（审批流）
- Budget & Cost Control（成本控制）
- Activity & Events（活动日志）
- Plugins（插件系统）
- Secrets & Storage（密钥存储）

### 新增 / 重写的模块

| 模块 | 说明 | 来源 |
|------|------|------|
| **HR Agent 引擎** | 需求分析 → 官网搜索 → 匹配推荐 → 成本判断 | 你的设计 |
| **Key 池独立管理** | Paperclip key 绑定在 agent 上，需分离 | 改造 |
| **Agent 池独立管理** | 同上，分离出独立池 | 改造 |
| **成本阈值确认机制** | HR 预估成本 → 超限等待确认 | 新增 |
| **项目工作空间** | 平等协作、互相 review 的讨论环境 | 新增 |
| **Dashboard 交互下钻** | 所有卡片可点击，逐层深入 | 改造 Paperclip UI |
| **移动聊天入口** | Telegram Bot + 聊天 Agent → 自然语言控制 | 参考 Mesa 实现 |
| **CLI 命令体系** | 统一按你的设计改造 | 改造 |

### 借鉴 Mesa 的设计哲学

- 保持调度路径简洁，不引入不必要的抽象层
- 所有调度决策在单一代码路径中完成，可追踪可审计
- Agent 之间不直接通信，通过调度器的任务/事件机制间接协作
- 成本作为第一公民，贯穿所有调度决策

## 范围边界（不在本次范围内）

- 非 LLM 调度器的稳定性保障（由程序保证）
- 多机分布式部署（初期为单机框架）
- 复杂权限管理（初期为单用户）
